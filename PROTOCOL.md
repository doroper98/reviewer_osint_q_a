# Reviewer ↔ OSINT 교신 프로토콜 (v1)

`agents_reviewer`(보고서·`report_bundle` **producer**)와 `osint_generator`(영상
**consumer**) 사이의 계약·질문·통지를 **비동기 파일 기반**으로 주고받는 규칙.
목적: 사람이 두 저장소 사이에서 메시지를 복붙 중계하지 않게 한다. 이 저장소가
교신의 SSOT(단일 출처)다.

## 참여자 (sender 코드)

| 코드 | 주체 |
|------|------|
| `agent_reviewer_bot` | agents_reviewer 쪽 에이전트 (질문·통지 발신) |
| `osint_generator`    | osint_generator 쪽 에이전트/담당자 (질문·통지 발신) |

양쪽 다 발신자가 될 수 있다. 규칙은 대칭이다.

## 메시지 종류 (kind) — 파일명 토큰

스레드는 **두 종류**다. 파일명·구조·상태 흐름이 다르다.

| kind | 토큰 | 언제 | 답변 의무 | 상태 흐름 |
|------|------|------|-----------|-----------|
| **질문 / 요청** | `q` | 상대의 답변·결정이 필요할 때 (묻기, 요청, 협의) | 필수 | `open → answered → closed` |
| **통지** | `n` | 일방 공지 — 계약 변경·릴리스·결정 사항 알림 | 선택(ack) | `posted → (acked) → closed` |

- **통지(n)** 는 답변을 강제하지 않는다. 다만 상대가 *동기화/확인*해야 하면
  (예: 양쪽 계약 문서 sync) `ack_required: yes` 로 표시하고, 상대는 `## 확인 ·
  동기화` 를 채운다. 순수 참고(FYI)면 `ack_required: no` — 발신자가 알림만 하고
  바로 `closed` 로 둘 수 있다.
- 판단이 애매하면: *상대가 뭔가 답하거나 결정해야 한다* → `q`. *이미 정해진 걸
  알리는 것* → `n`.

## 핵심 규칙

1. **한 사안 = 한 스레드 파일.** osint(또는 reviewer)에게 전달·질문·통지할 게
   생기면 **항상 새 파일**을 만든다. 기존 파일에 다른 사안을 얹지 않는다.
2. **파일명 코드 체계** — `yyyy_mm_dd_hhmmss_<sender>_<kind>_nn.md`
   - `<sender>` = `agent_reviewer_bot` 또는 `osint_generator` (발신자).
   - `<kind>` = `q`(질문·요청) 또는 `n`(통지).
   - `nn` = 발신자·kind 별 2자리 일련번호(`01`, `02`, …). (발신자마다, q/n 각각 독립 카운터.)
   - `yyyy_mm_dd_hhmmss` = 생성 시각(KST).
   - 예: `2026_07_09_110842_agent_reviewer_bot_n_01.md` (통지),
     `2026_07_09_143000_osint_generator_q_01.md` (질문).
3. **파일 위치** — 모든 스레드는 `threads/` 디렉토리에 둔다.
4. **한 파일 = 한 스레드.** 같은 파일 안에서 종류별 구조로 이어 쓴다.
   - **질문(q)**: 발신자 `## 질문 / 전달` → 수신자 `## 답변` → 발신자
     `## 조치 · 종결`(무엇을 어떻게 조치했고 언제 종결했는지) + `status: closed`.
   - **통지(n)**: 발신자 `## 통지` → (ack 필요 시) 수신자 `## 확인 · 동기화` →
     발신자 `## 종결` + `status: closed`. FYI(ack 불요)면 발신자가 `## 통지`
     작성 후 바로 `closed` 가능.
5. **답변·확인에 새 요청을 끼워넣지 않는다.** 답변·확인·종결을 쓰다가 새로 부탁·
   질문·통지할 게 생기면, 거기에 넣지 말고 **새 스레드 파일**(규칙 1·2)을 만든다.
   한 스레드는 원래의 한 사안만 다룬다. (교차 참조는 `refs` 로 건다.)
6. **상태(status) 라이프사이클** — 헤더 메타의 `status` 를 kind 에 맞게 갱신한다:
   - 질문(q): `open`(답변 대기) → `answered`(답변 옴, 조치 중) → `closed`.
   - 통지(n): `posted`(공지됨) → `acked`(상대 확인·동기화 완료, ack 필요 시) →
     `closed`(발신자 종결). ack 불요 FYI 는 `posted → closed`.
   - 필요 시 `reopened` 후 위 흐름 반복.
7. **교차 참조(`refs`)** — 다른 스레드/계약 조항/버전과 얽히면 파일명·`§조항`·
   버전(예: `IMAGE_BUNDLE_CONTRACT §3.1-a`, `producer v8.3.6`)을 `refs` 에 적는다.
8. **커밋** — 각자 자기 쪽 변경만 커밋·푸시한다. 커밋 메시지에 스레드 코드
   (`<kind>_nn`)를 포함한다. 상대 스레드 파일을 임의로 수정하지 않는다(자기 섹션만).
9. **한 사안 종결의 정의** — 발신자가 (질문이면 답변을, 통지면 필요한 ack 를) 받아
   실제 조치(코드/문서/버전 반영 또는 "조치 불요" 판단)를 완료하고 종결 섹션 +
   `status: closed` 를 기록한 시점. 답변·ack 만 오고 종결이 없으면 스레드는 열려 있다.

## 파일 템플릿

- 질문·요청 → `templates/question_template.md`
- 통지 → `templates/notice_template.md`

복사해 파일명 규칙(규칙 2)대로 저장한 뒤 채운다.

## 헤더 메타 필드

| 필드 | 의미 |
|------|------|
| `id` | 파일명(확장자 제외)과 동일한 스레드 코드 |
| `kind` | `question` / `notice` |
| `from` / `to` | 발신자 / 수신자 코드 |
| `created` | 생성 시각 (KST) |
| `status` | q: `open`/`answered`/`closed` · n: `posted`/`acked`/`closed` (+`reopened`) |
| `ack_required` | 통지(n) 한정 — `yes`/`no`. 질문(q)은 `-` |
| `refs` | 관련 계약 조항·버전·다른 스레드 파일명 |

## 이 프로토콜 자체의 변경

이 문서(`PROTOCOL.md`)가 규칙의 SSOT다. 변경 시 `agents_reviewer` 의
`CLAUDE.md` "OSINT 교신 규칙" 섹션과 **정합**을 맞춘다(양쪽에 요약이 있으나 상세
규칙의 정본은 이 파일). 규칙 변경도 하나의 스레드로 제안·합의 후 반영한다.
